# RAG 查询改写：三种方法 + 面试标准回答

## 📌 一句话总结

查询改写的本质是对原始提问做语义增强。策略是**规则兜底 + 小模型批量 + 大模型处理复杂语义**，让模糊问题标准化，提升召回和准确率。

---

## 🎯 为什么要做查询改写？

```
用户输入："怎么报销"（模糊、口语化、多义）
        ↓ 查询改写（桥）
系统理解：报销流程？报销入口？差旅标准？项目经费？
        ↓
匹配知识库：《差旅费用报销流程 v3.2》
```

> 用户说的随意，知识库很专业。中间的桥就是查询改写。做不好 → 召回偏 → 整个 RAG 链条断在最前面。

---

## 🔧 三种方法（面试按这个顺序讲）

### 方法一：基于规则

| 维度 | 说明 |
|------|------|
| 做法 | 关键词映射、同义词表、词表替换 |
| 优点 | 快速上线，固定问法效果好 |
| 缺点 | 泛化弱，换业务规则全废 |
| 定位 | **兜底方案** |

### 方法二：基于 NER 模型

| 维度 | 说明 |
|------|------|
| 做法 | 小模型做实体抽取 + 同义词替换 |
| 优点 | 泛化比规则好，速度快 |
| 缺点 | 需要标注数据，换业务要重训 |
| 致命伤 | 不懂深层语义，跨轮指代直接废 |

> 例：用户问"他是谁"，上一句说"马斯克"→ NER 无法跨轮理解

### 方法三：基于大模型（主流，面试官最想听）

#### 能力一：跨轮语义补全

```
用户："他是谁"（上一句提到"马斯克"）
→ 大模型改写："马斯克是谁"
→ 传统 NER 做不到
```

#### 能力二：多路扩展改写

```
用户："AI产品经理要会什么"
→ 大模型同时生成：
  "AI产品经理核心技能要求"
  "AI产品经理能力模型"
  "AI产品经理需要掌握的技术栈"
→ 召回率拉满，法务/医疗领域特别好用
```

#### 能力三：实体标准化

```
用户："豆包是字节的吗"
→ 改写："豆包是否由字节跳动开发"
→ 匹配知识库中的正式文档
```

| 优点 | 不需要训练数据，写好 Prompt 直接上线 |
|------|------|
| 缺点 | 成本高、速度慢 |

---

## 🎤 面试标准回答

> "查询改写的本质是对原始提问做语义增强。我的策略是三层互补：
>
> **规则兜底**：固定高频问法走规则，保证快速响应。
> **小模型做批量**：NER 做快速改写，处理量大的简单场景。
> **大模型处理复杂语义**：跨轮补全、多路扩展、实体标准化，处理真正难的问题。
>
> 三层配合，让模糊问题标准化，提升召回和准确率。这就是 RAG 落地的核心。"
