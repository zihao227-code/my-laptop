# 05 · Prompt 迭代优化方法

## Prompt 不是一次写完的

你写需求文档也不是一次过的——要评审、要改、要迭代。Prompt 也一样。**写完 → 跑一遍 → 看输出哪里不满意 → 改 → 再跑**，两三轮后基本稳定。

## 最常见的三个错误

### 错误 1：太模糊

```
❌ "帮我优化一下这个需求"
```
什么叫优化？改格式还是改逻辑？增加细节还是删减冗余？换一个人，理解完全不一样。

```
✅ "帮我检查这个需求文档是否有遗漏的异常流程。
   用清单列出所有可能出错的场景，每个场景标注对应的兜底方案。"
```

**改法**：把你脑子里"优化"的具体含义全部写出来。AI 没有读心术。

### 错误 2：一次给太多

```
❌ "帮我分析需求、画流程图、写用户故事、排优先级"
```
这一个 Prompt 里塞了四件完全不同的事。AI 会做，但每件事的深度都会打折。

```
✅ 拆成四个请求：
   1. 分析需求 → 输出功能点清单
   2. 基于功能点 → 画流程图
   3. 基于功能点 → 写用户故事
   4. 基于功能点和依赖关系 → 排优先级
```

**改法**：一个 Prompt 只做一件事。像拆需求一样拆 Prompt。

### 错误 3：不给约束

```
❌ "写一段产品介绍"
```
字数、风格、目标受众——AI 全不知道。它可能给你 200 字的简版，也可能给你 2000 字的详版。

```
✅ "写一段 200 字以内的产品介绍。目标受众是公司高层，
   突出商业价值，不要技术细节。语气要正式但不生硬。"
```

**改法**：把你心里的"验收标准"全部写出来。你不说，AI 不会猜。

## 迭代三步法

```
第1轮：写 Prompt → 看输出
       ↓
       标记哪里不满意（格式？深度？语气？遗漏？）
       ↓
第2轮：修改 Prompt → 再跑
       ↓
       对比两轮输出，确认改对了方向
       ↓
第3轮：微调 → 跑最后一次
       ↓
       满意 → 给 Prompt 标版本号 V1.0，存下来
```

## 版本管理：像管代码一样管 Prompt

AI PM PRD 八步法里有一条铁律：**Prompt 必须做版本管理**。

你在终端用 Git，Prompt 也可以用同样的方式：

```
Prompt文件/提取功能点_V1.0.md
Prompt文件/提取功能点_V1.1.md  （加了一个范例）
Prompt文件/提取功能点_V2.0.md  （重构了输出格式）
```

每次改 Prompt，记三句话：
1. 改了什么
2. 为什么改
3. 改了之后效果变好了还是变差了

这花不了 30 秒，但两个月后你回头看，能清楚地追溯你是什么时候、为什么把某句话加进 Prompt 的。
