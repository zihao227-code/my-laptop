# AI 辅助开发的工程质量

## 1. 上下文管理

### 问题
跟 AI 聊太久，对话历史塞满上下文窗口，怎么办？

### Claude Code 的机制
- 对话长度达到 **80-95%** 时自动触发 `/compact`（压缩），将历史对话提炼成摘要
- 达到 **100%** 时强制截断，前面丢失（除非已被 compact）

### 主动管理手段
| 手段 | 说明 |
|------|------|
| `/compact` | 手动触发压缩 |
| 写 `notes/` | 每个主题学完后沉淀笔记 |
| 写 `.claude/memory/` | 长期有效的决策和偏好 |
| `CLAUDE.md` | 项目级固定上下文 |
| `MEMORY.md` | memory 文件索引 |

## 2. 信息提炼与检索

### 分层存储体系
| 存储位置 | 适合内容 | 生命周期 |
|----------|---------|---------|
| `notes/` | 学习笔记、概念理解 | 长期 |
| `.claude/memory/` | 长期决策、用户偏好 | 长期 |
| `CLAUDE.md` | 项目描述、学习目标 | 长期 |
| `MEMORY.md` | memory 索引 | 长期 |

### 快速定位方法
- 关键词搜索 `notes/` 目录
- 查看 `MEMORY.md` 索引
- `git log` 查看每次 commit 对应的话题
- **缺失：学习日志**（按时间线记录每次学习内容）

## 3. 回归测试（Regression Testing）

### 核心概念
新代码提交后，重新跑一遍已有测试，确保旧功能不受影响。

### 教育平台的测试体系
```
tests/
├── e2e/           ← Playwright 自动化测试
├── run-tests.js   ← 一键跑全部
└── results/
    └── latest.json ← 测试结果
```

### 理想工作流
```
改代码 → 跑全部测试 → 全部通过 ✅ → 放心提交
                    → 有失败 ❌ → 定位 → 修复 → 再跑
```

## 4. 变更影响分析（Impact Analysis）

### 动代码前三个问题
1. **这个文件/函数被谁依赖？** — 改了 login，所有要用登录的页面都受影响
2. **是"加东西"还是"改行为"？** — 加新功能影响小，改已有行为影响大
3. **数据格式变了吗？** — 改了存储结构？旧数据还能读吗？改了参数？调用方要改吗？

### 防错手段排序
| 优先级 | 手段 | 成本 | 效果 |
|--------|------|------|------|
| 🥇 | 手动点一遍关联页面 | 低 | 覆盖常见路径 |
| 🥈 | 跑 Playwright 自动化测试 | 中 | 覆盖关键链路 |
| 🥉 | 改前画依赖图 | 低 | 防止遗漏影响范围 |
