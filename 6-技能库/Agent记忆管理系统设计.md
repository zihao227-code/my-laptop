# Agent 记忆管理系统设计

## 📌 一句话总结

记忆本质上不是存数据，而是帮 Agent 把原始交互沉淀为可调用的经验和可执行的知识。向量库 + RAG 只是入门，生产级系统需要分层架构 + 工程骨架 + 时间建模 + 经验固化。

---

## 🎯 面试考察本质

> 字节 AI 岗真题：「你们 Agent 是怎么做记忆管理的？」

| 考察层次 | 真正问什么 |
|------|------|
| 第一层 | 你知不知道向量库 + RAG 的**边界**在哪？ |
| 第二层 | 你有没有做**分层记忆**设计？ |
| 第三层 | 你有没有处理过记忆在**时间维度上的状态混乱**？ |

---

## ❌ 入门级回答（Demo 阶段）

```
"用向量数据库存历史对话，embedding 后 RAG 捞出来拼到 prompt 里"
```

### 两个生产级隐患

| 隐患 | 表现 | 根因 |
|------|------|------|
| **精确查询失效** | 查手机号/订单号，返回语义相近片段但拿不到精确值 | 向量库只做语义匹配，不做键值查找 |
| **时间盲区** | 用户先说"在上海"后说"搬到北京"，两条记录语义相近被同时捞出，Agent 分不清当前状态 | 向量库不建模时间维度 |

---

## ✅ 生产级设计

### 一、四层记忆架构

```
┌─────────────────────────────────────┐
│ 第四层：滑动窗口                     │
│ 当前交互原始上下文，超窗口即丢弃       │
│ 典型 LLM 短期记忆                   │
├─────────────────────────────────────┤
│ 第三层：近期对话摘要                 │
│ 完整对话→压缩为主题+意图+关键实体     │
│ 保留语义走向，压缩 Token            │
│ ← 语义检索在这一层使用               │
├─────────────────────────────────────┤
│ 第二层：用户结构化档案卡             │
│ 身份/等级/偏好/语言/历史决策         │
│ JSON/关系型存储，精确键值查找         │
│ ← 绝不经过语义检索                   │
├─────────────────────────────────────┤
│ 第一层：会话元数据                   │
│ 会话ID/入口渠道/临时标记             │
│ 用完即弃，纯内存，不持久化           │
└─────────────────────────────────────┘
```

**核心原则**：语义检索只在摘要层等少数场景使用。精确读取和状态维持由结构化档案卡承担。**不能让向量库一统天下**。

---

### 二、工程骨架：Ledger / Views / Policy

| 组件 | 职责 | 关键特性 |
|------|------|------|
| **Ledger（账本）** | 只追加、不可修改的原始事件流 | 用户说过什么、系统执行了什么，单向追加，源头数据永不丢失，任何决策可回溯 |
| **Views（视图）** | 面向任务的派生视图 | 把 Ledger 原始事件流按当前任务聚合为可读结构。不同任务拉不同 Views |
| **Policy（策略）** | 读写和遗忘的控制策略 | 规定什么时候写记忆、什么时候读记忆、什么时候遗忘 |

#### 遗忘与隐私的处理

用户要求删除信息时 → **不删 Ledger**，而是通过 Policy 控制 Views 过滤掉对应数据，同时保留可审计的痕迹。

---

### 三、时间状态冲突：双时间戳机制

每条事实带两个时间戳：

| 时间戳 | 含义 |
|------|------|
| **Valid Time** | 这条信息在现实世界里的有效时段 |
| **Transaction Time** | 它写入系统的时间 |

**示例：用户从上海搬到北京**

```
不是：往向量库新增一条语义相似记录
而是：
  1. 把"居住地=上海"的 Valid Time 截止到搬家时刻
  2. 插入"居住地=北京"（当前有效）

读取时始终按 Valid Time 过滤 → 新旧信息不再互相污染
```

---

### 四、经验固化：程序性记忆（Skills）

当 Agent 多次成功完成某类任务（如处理退款申诉）：

```
完整交互轨迹 + 决策路径
        ↓
  沉淀为可复用的 Skill
（标准步骤 + 必要参数 + 校验规则）
        ↓
  下次同类任务直接加载执行
        ↓
  Agent 越来越像熟练员工
```

本质：把高频经验固化成内部流程资产，而不是每次从零开始推理。

## 🎤 面试回答策略

### 递进式五步回答

| 步骤 | 内容 | 目的 |
|------|------|------|
| **1. 承认常规** | 向量库+RAG 是常规手段 | 展示你懂基础 |
| **2. 指出局限** | 精确查询失效 + 时间盲区 | 展示你有生产经验 |
| **3. 展示架构** | 四层分层设计，各层读写策略不同 | 展示你有系统设计能力 |
| **4. 补上骨架** | Ledger + Views + Policy 工程三件套 | 展示你考虑到了工程深度 |
| **5. 提升高度** | 双时间戳解决时间冲突 + Skill 经验固化 | 展示你能设计进化型系统 |

## 🆚 对比总结

| 维度 | Demo 做法 | 生产做法 |
|------|------|------|
| 存储 | 一个向量库 | 四层分层（元数据/档案/摘要/窗口） |
| 精确查询 | 语义检索一把梭 | 键值查找 + 语义检索各司其职 |
| 时间处理 | 无时间建模 | 双时间戳（Valid Time + Transaction Time） |
| 遗忘 | 物理删除 | Ledger 不可变 + Views 过滤 |
| 进化 | 每次从零推理 | 高频路径沉淀为 Skill |
