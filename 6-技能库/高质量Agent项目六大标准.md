# 高质量 Agent 项目六大标准

## 📌 一句话总结

不是接模型 + 写 Prompt + 调工具 + 做页面就是 Agent。能写进简历的项目 = **真实业务 × 工程体系 × 智能决策 × 可观测 × 人机协同**。

---

## ❌ 反面：面试官为什么不要你？

```
Demo 级 Agent：
  接一个模型 → 写几个 Prompt → 调几个工具 → 做一个页面
  用户输入一句话 → 模型回复一句话

面试官判定：
  这是聊天机器人，不是 Agent
  没有业务价值、没有工程能力
```

---

## ✅ 六大标准

### 标准一：必须解决真实问题

| ❌ | ✅ |
|------|------|
| 为了 Agent 而 Agent | 智能客服、企业知识库、旅行规划、销售助手 |
| 用户输入 → 模型回复 | 明确用户需求 + 完整业务流程 + 能评估效果 |

> 把你的 Agent 放回业务里，它帮谁省了多少时间、少出了多少错？

### 标准二：完整的后端工程体系

| Demo | 企业级 |
|------|------|
| 只有 Prompt 和调用链 | API 接口 + 数据库设计 |
| 本地跑录个屏 | 权限管理 + 日志追踪 |
| 单次请求返回 | 异常处理 + 多用户并发 |

> 只有 Prompt 没有工程架构 = Demo。

### 标准三：Agent 真正的核心能力

Agent ≠ 用户提问 → 模型调用工具 → 返回答案

必须体现：
- **任务规划**：任务怎么拆？下一步做什么？调用哪个工具？
- **失败处理**：工具调用失败后怎么降级？怎么重试？
- **结果验证**：返回结果对不对？谁来验证？
- **多 Agent 协同**：复杂任务拆分给多个子 Agent

> 注意：不要为了炫技做多 Agent。架构匹配业务，需要几个就几个。

### 标准四：上下文工程

> 上下文 ≠ 保存聊天记录

真正的上下文设计：**让模型在正确的时候看到正确的信息**

| 上下文类型 | 使用时机 |
|------|------|
| 当前任务上下文 | 跟当前任务相关的信息 |
| 用户历史记忆 | 用户之前说过什么、偏好什么 |
| 企业知识库 | 跟当前问题相关的业务知识 |
| 工具调用结果 | 刚才调工具返回了什么 |

→ 直接关联 [[Agent记忆管理系统设计]] 的四层记忆架构

### 标准五：可观测性和评估体系

> 不能靠一句"我感觉效果挺好"

| 追踪维度 | 具体问题 |
|------|------|
| 调用链路 | 调了哪个 Agent？哪个模型？哪个工具？ |
| 执行状态 | 成功了？失败在哪一步？ |
| 性能指标 | 响应速度？成本？ |
| 结果质量 | 结果准确吗？用户满意吗？ |

→ 直接关联 [[企业级AI开发四大核心能力#3. 评测和可观测性]]

### 标准六：人机协同

> 不要认为最好的 Agent 就是完全自动化

保留人工干预的场景：
- 发送邮件（确认内容和收件人）
- 提交审批（关键决策需要人签字）
- 资金操作（高风险，必须人确认）
- 隐私数据（合规要求）
