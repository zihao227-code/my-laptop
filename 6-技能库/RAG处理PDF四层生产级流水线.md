# RAG 遇到 PDF：四层生产级处理流水线

## 📌 一句话总结

把 PDF 处理看作一条多模态文档理解流水线：身份识别 → 骨架还原 → 语义切片与索引 → Agent 工具集。不是把 PDF 当纯文本流暴力处理。

---

## ❌ 面试低分回答

> "把 PDF 文本抽出来，切一切，灌进向量库。"

| 理想 PDF | 真实业务 PDF |
|------|------|
| 纯文字、排版规整 | 多栏、表格、扫描件、页眉广告、手写签名 |
| 抽文本即可 | 搜出来全是缺头少尾的碎片垃圾 |

---

## 🔄 四层生产级流水线

### 第一层：身份识别

| PDF 类型 | 本质 | 处理方式 |
|------|------|------|
| 原生文本 PDF | 文字隐藏，用文本指令画 | 解析器精准提取 + 保留位置 |
| 扫描件 PDF | 图片容器，没有字符 | OCR（Tesseract / 文档识别模型） |
| 图文混排 PDF | 正文嵌入票据、架构图、签章 | 多模态模型端到端视觉理解 |

> Anthropic Claude PDF Support：不是解析 PDF 结构，而是像人眼一样直接阅读每一页的视觉呈现，把图像和文本作为整体来认知。

### 第二层：骨架还原

```
标题层级 → 文字坐标 + 字体大小推断
表格行列 → 线条 + 空白区还原
页眉页脚 → 识别后剔除或标记为 Metadata
章节边界 → 段落关系建模
```

**为什么这一步致命**：合同条款 → 必须追溯到第几条第几页。财报数字 → 必须知道出自哪张表、哪个财年。缺这层，回答无法验证，风控/合规场景致命。

### 第三层：语义切片与索引

| ❌ 固定长度 | ✅ 语义单元 |
|------|------|
| 512 token 一刀切 | 同标题段落 → 一个 Chunk |
| 表格拦腰截断 | 完整表格 → 单独结构化 |
| 图文拆散 | 图 + 说明文字 → 绑定 |

**每个 Chunk 携带**：文档 ID、页码、章节标题、表格编号、时间戳、版本号。

**Contextual Retrieval（Anthropic）**：索引时主动补上下文说明：
```
不是只存："汇率波动导致海外营收下降 15%"
而是加上："该片段出自第三章风险因素，讨论汇率波动对海外营收的影响"
→ 检索命中时，模型看到的是带着位置感的信息单元
```

### 第四层：Agent 工具集

> 不是检索出 Chunk 后全塞进 Prompt = 暴力阅读。

| 工具 | 功能 |
|------|------|
| `search_pdf` | 语义检索相关片段 |
| `read_page` | 直接读取指定页全文 |
| `extract_table` | 抽表格转 CSV/Markdown |
| `analyze_chart` | 视觉模型看图说话 |
| `source` | 返回带页码和片段的引用 |

```
简单问题 → 一次 search_pdf
复杂问题 → Agent 规划多步：
  搜索章节 → 抽表格 → 综合比较 → 视觉解析图表
→ 动态按需，不搬整座文档山
```

---

## 🏗️ 第五层：引用 + 评测闭环

| 层次 | 做法 |
|------|------|
| 引用 | 每个论断标注页码、段落、原文片段 |
| 评测 | 拆分量化每个环节 — Chunk 来自正确页码？表格行列值准确？OCR 漏字？图表理解正确？ |

---

## 🎤 面试要点

> 面试官真正考察的是：面对非结构化数据时，有没有自底向上的工程拆解能力和从能跑到可靠的架构意识。你能把 PDF 这道脏活累活拆明白、讲通透，就意味着有底气接住真正落地的项目。
