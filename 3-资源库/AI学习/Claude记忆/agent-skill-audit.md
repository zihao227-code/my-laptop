对 `.claude/agents/` 下的 6 个 Agent 进行实用性审计，结合用户的实际情况（零基础 BA、学习 AI Coding、主要通过主对话交互）。

## 审计结论总览

| Agent | 模型 | 实用性 | 建议 |
|-------|------|--------|------|
| BA导师 | sonnet | ⭐⭐⭐⭐⭐ 核心 | 保留，最匹配用户身份 |
| 测试大师 | sonnet | ⭐⭐⭐ 有场景 | 保留，跑测试时用 |
| 技术执行者 | haiku | ⭐⭐ 偶尔 | 保留，批量化代码执行场景 |
| 方案审视者 | sonnet | ⭐⭐ 低频 | 可精简或合并到主对话 |
| 问题解决方案探索者 | sonnet | ⭐⭐ 低频 | 主对话已覆盖大部分场景 |
| Agent管家 | sonnet | ⭐ 用不上 | 过度设计，可降级为 CLAUDE.md 规则 |

## 逐个分析

### ✅ BA导师 — 核心保留
- **为什么重要**：用户是 BA，需要 BA 视角的讲解和引导
- **典型场景**：解释技术概念（用 BA 语言）、审视需求、学习辅导
- **为什么不用主对话代替**：BA 视角需要专门的 system prompt 约束，主对话容易滑向纯技术讲解
- [[vibe-coding-concepts]] 中的概念学习主要靠这个 Agent

### ✅ 测试大师 — 有明确场景
- **使用场景**：教育平台测试用例撰写、测试运行、Bug 分析
- **触发条件明确**：任何时候涉及测试相关任务
- **与 [[edu-platform-starter]] 紧密配合**

### ⚠️ 技术执行者 — 保留但低频
- **使用场景**：批量文件操作、配置安装、后台代码任务
- **使用 haiku 模型降低成本** — 这是保留的主要理由
- **注意**：用户不需要主动调用，由主对话在需要时自动分配

### ⚠️ 方案审视者 — 可被主对话替代
- **问题**：审查方案这件事，主对话本身就能做（且一直在做）
- **唯一不可替代的场景**：需要独立的"第二意见"时（adversarial review）
- **建议**：保留文件但不主动触发，只在需要独立审查时手动调用

### ⚠️ 问题解决方案探索者 — 可被主对话替代
- **设计意图**：先搜索网络 → 对比方案 → 推荐最优
- **实际使用**：主对话遇到问题时直接用 WebSearch + 分析，不需要额外 Agent 封装
- **建议**：降级为一段方法论文档（"遇到问题时的工作流程"），而非独立 Agent

### ❌ Agent管家 — 过度设计
- **设计意图**：管理 7+ Agent 的复杂生态系统、冲突检测、合并分析
- **实际情况**：用户只有一个主对话 + 偶尔调用的 2-3 个 Agent，不存在需要"管家"管理的复杂度
- **具体问题**：
  - 拓扑图维护成本高（每次改 Agent 都要更新图）
  - 自身消耗 token（审查报告、路由建议都是消耗）
  - 对于 6 个 Agent 的生态来说没有实际价值
- **建议**：保留文件作为 Agent 设计原则的参考文档，但不作为活跃 Agent 使用。在 CLAUDE.md 加一条简单规则即可替代路由功能

## 实际使用的 Agent 调用模式

大多数任务由**主对话直接处理**，以下场景才调用子 Agent：

1. **概念学习 / BA 视角讲解** → BA导师
2. **测试相关（写用例、分析 Bug、跑测试）** → 测试大师
3. **批量代码执行** → 技术执行者（haiku，省钱）

其他三个 Agent 基本不会主动调用。

**Why:** 用户的学习场景不需要多 Agent 编排。简洁的 Agent 体系 > 完整的 Agent 体系。当前 6 Agent 偏多，实际活跃的只有 3 个。

**How to apply:** 新 Agent 创建前先问——主对话能不能做？如果主对话能做，不创建新 Agent。对现有 Agent 不做删除（保留文件），但实际使用中只活跃 3 个核心 Agent。
